Create `src/security/command-classifier.ts` with these exact exports:

```typescript
export type CommandClassification = "blocked" | "risky" | "safe";
export function classifyCommand(command: string, args: string[]): CommandClassification;
```

**BLOCKED_PATTERNS** (RegExp array) - test against full invocation string `[command, ...args].join(" ")`:
- `/^rm\s+.*-rf\s+\/($|\s)/` and `/^rm\s+.*\/($|\s).*-rf/` — rm -rf / in either arg order
- `/^mkfs/` — any mkfs variant
- `/^dd\s+if=\/dev\/(zero|random)/` — dd from device files
- `/\|\s*(bash|sh|zsh)/` — pipe to shell (curl|sh pattern)
- `/^sudo\s+su/` — privilege escalation
- `/^chmod\s+.*-R.*777.*\//` — recursive 777 on root paths
- `/^:\(\)\{.*\|.*:\&\}/` — fork bomb

**SAFE_COMMANDS** (Set of strings): `ls`, `cat`, `git`, `grep`, `pwd`, `echo`, `head`, `tail`, `wc`, `find`, `which`, `env`, `date`, `ps`, `df`, `du`, `uname`, `whoami`, `hostname`, `uptime`, `history`, `type`, `file`, `stat`, `less`, `more`, `sort`, `uniq`, `cut`, `tr`

**DANGEROUS_FLAGS** (strings to check in args): `--exec`, `-exec`, `--delete`, `--write`, `-i` (for find -exec, etc.)

**SCRIPT_EXTENSIONS**: `.sh`, `.py`, `.ts`

Classification logic:
1. Build `fullInvocation = [command, ...args].join(" ")`
2. If any BLOCKED_PATTERN matches fullInvocation -> return `"blocked"`
3. Extract basename: `command.split("/").pop() ?? command`
4. If command is a script (ends with .sh, .py, .ts) -> return `"risky"` (per CONTEXT.md: scripts always risky)
5. If basename is in SAFE_COMMANDS:
   - If any arg is in DANGEROUS_FLAGS -> return `"risky"`
   - Otherwise -> return `"safe"`
6. Default (unknown command) -> return `"risky"` (fail-closed per CONTEXT.md)

Also export a helper for getting the block reason:
```typescript
export function getBlockReason(command: string, args: string[]): string | null;
```
Returns a human-readable reason string if blocked (e.g., "Recursive deletion of root filesystem is not allowed"), or null if not blocked. This is used by the tool to explain WHY a command was blocked.

**Part B: Create `src/security/approval-gate.ts`**

Exports:
```typescript
export interface ApprovalGate {
  request(params: ApprovalRequest): Promise<boolean>;
  handleCallback(id: string, approved: boolean): void;
  recoverPendingOnStartup(notify: (userId: string, message: string) => Promise<void>): void;
}

export interface ApprovalRequest {
  command: string;
  args: string[];
  cwd: string;
  reason: string;
  userId: string;
  sendApproval: (text: string, approvalId: string) => Promise<void>;
}

export function createApprovalGate(db: Database.Database): ApprovalGate;
```

Implementation details:

1. Import `randomUUID` from `node:crypto`, import `Database` type from `better-sqlite3`, import `log` from `../logger.js`.

2. Module-level `resolvers` Map: `Map<string, (approved: boolean) => void>` — stores in-memory Promise resolvers keyed by approval ID.

3. `request(params)` method:
   - Generate `id = randomUUID()`
   - Calculate `expiresAt = new Date(Date.now() + 5 * 60 * 1000).toISOString()`
   - **Write to SQLite FIRST** (survives restart): INSERT INTO pending_approvals with id, user_id, command, JSON.stringify(args), cwd, reason, 'pending' status, expiresAt
   - Call `params.sendApproval(message, id)` where message includes: the command + args, cwd, reason, expiry time
   - Return `new Promise<boolean>((resolve) => { ... })`:
     - Store resolver in `resolvers` Map keyed by `id`
     - Set 5-minute timeout: if resolver still in Map, delete it, UPDATE pending_approvals SET status='expired' WHERE id=?, resolve(false)
     - Log timeout event

4. `handleCallback(id, approved)` method:
   - Look up resolver in Map
   - If found: delete from Map, UPDATE pending_approvals SET status = (approved ? 'approved' : 'denied') WHERE id = ?, call resolver(approved)
   - If NOT found (post-restart case): just UPDATE pending_approvals SET status WHERE id = ? (no Promise to resolve)
   - Log the decision

5. `recoverPendingOnStartup(notify)` method:
   - SELECT * FROM pending_approvals WHERE status = 'pending'
   - For each row: UPDATE status to 'expired', call notify(row.user_id, message) where message explains the command expired during restart
   - Log recovery count

All SQLite operations use `db.prepare(...).run(...)` (synchronous, per better-sqlite3 pattern).
