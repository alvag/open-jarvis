```typescript
const createScheduledTaskTool: Tool = {
  definition: {
    name: "create_scheduled_task",
    description: "Create a scheduled task or reminder. Extract cron expression from user's natural language before calling. Use 'reminder' type for simple text notifications (no agent loop). Use 'task' type for tasks that need tool execution.",
    parameters: {
      type: "object",
      properties: {
        name: { type: "string", description: "Human-readable task name, e.g. 'Daily standup reminder'" },
        type: { type: "string", description: "Task type", enum: ["reminder", "task"] },
        cron_expression: { type: "string", description: "Cron expression (5 fields: min hour day month weekday) OR ISO 8601 datetime for one-shot tasks, e.g. '0 9 * * 1' for every Monday 9am, or '2026-03-19T09:00:00' for a specific time" },
        prompt: { type: "string", description: "For reminders: the notification text. For tasks: the instruction for the agent to execute" },
        timezone: { type: "string", description: "IANA timezone, e.g. 'America/Bogota'. Defaults to system timezone" },
      },
      required: ["name", "type", "cron_expression", "prompt"],
    },
  },
  async execute(args, context): Promise<ToolResult> {
    try {
      const task = createTask({
        userId: context.userId,
        name: args.name as string,
        type: args.type as TaskType,
        cronExpression: args.cron_expression as string,
        prompt: args.prompt as string,
        timezone: (args.timezone as string) || config.scheduler.timezone,
      });
      const nextRun = getNextRun(task.id);
      return {
        success: true,
        data: {
          id: task.id,
          name: task.name,
          type: task.type,
          cron_expression: task.cron_expression,
          next_run: nextRun?.toISOString() ?? "N/A",
        },
      };
    } catch (err) {
      return { success: false, data: null, error: (err as Error).message };
    }
  },
};
```

## Scheduled Tasks
When users ask you to schedule something:
- Extract: what to do (prompt), when (cron expression), task name

- Use type "reminder" for simple notifications, "task" for things that need tool execution
- Always confirm back: "Tarea creada: [name] — proxima ejecucion: [datetime]"
- For "show my tasks" or "list tasks": use list_scheduled_tasks
- For delete requests: ALWAYS confirm with user first ("Quieres que elimine la tarea '[name]'?"), then call delete_scheduled_task
- For pause/resume: use manage_scheduled_task

## Knowledge

Implementation:
- Create a BitbucketClient instance
- Prepare statements:
  - `selectPrState`: `SELECT * FROM pr_states WHERE pr_id = ?`
  - `upsertPrState`: `INSERT INTO pr_states (pr_id, workspace, repo_slug, last_updated_on, last_state, last_commit_hash, participant_states, checked_at) VALUES (?, ?, ?, ?, ?, ?, ?, datetime('now')) ON CONFLICT(pr_id) DO UPDATE SET last_updated_on = ?, last_state = ?, last_commit_hash = ?, participant_states = ?, checked_at = datetime('now')`
- Wrap entire body in try/catch — on error, log but do NOT throw (PR monitor failures should not crash the scheduler)
- List open PRs: `await client.listPRs(undefined, undefined, "OPEN")`
- Filter to PRs where user is author OR reviewer:
  - Author: `pr.author.nickname === config.bitbucket.email.split("@")[0]` (best-effort match)
  - Reviewer: `pr.reviewers.some(r => r.nickname === config.bitbucket.email.split("@")[0])`
  - NOTE: This is a heuristic per Claude's Discretion — exact matching depends on Bitbucket account setup
- For each relevant PR:
  - Get known state from `selectPrState.get(pr.id)` as `PrStateRow | undefined`
  - If no known state: INSERT baseline row via upsertPrState, continue (don't notify on first sight)
  - If `pr.updated_on !== known.last_updated_on`:
    - **Update pr_states BEFORE sending notification** (per pitfall 5 in RESEARCH.md — prevents duplicates)
    - Fetch activity: `await client.getPRActivity(String(pr.id))`
    - Build notification by analyzing activity items:
      - Count new commits: look for activity items of type update where `update.state === pr.state` (state unchanged = likely commit push)
      - Detect state changes: compare `pr.state` vs `known.last_state`
      - Detect direct mentions: check comment items for `@${config.bitbucket.email.split("@")[0]}` in content.raw
    - Update pr_states with new updated_on, state, etc.
    - Build notification message string:
      ```
      🔀 PR #${pr.id} "${pr.title}":
      [list changes — e.g., "2 new commits by Juan", "Estado: OPEN -> MERGED", "@you mentioned in comment"]
      ${pr.links.html.href}
      ```
    - Only send notification if there are actual reportable changes (not empty)
    - `await sendMessage(userId, notification)`
- Also check for recently merged/declined PRs that we were tracking:
  - List PRs with state MERGED or DECLINED: `await client.listPRs(undefined, undefined, "MERGED")`
  - For each, check if it's in pr_states with a different last_state
  - If state changed: update pr_states + notify
- Log: `log("info", "pr-monitor", "PR check complete", { prsChecked: count })`
